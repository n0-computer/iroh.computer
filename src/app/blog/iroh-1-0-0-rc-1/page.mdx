Welcome to `iroh@1.0.0-rc.1`, the last release candidate before 1.0.

Since [the first release candidate](/blog/iroh-1-0-0-rc-0) two weeks ago we collected feedback from you, fixed a bugs, and made the (hopefully) last API adjustments we wanted to make before stabilizing. Unless something serious comes up, the next release is **1.0**.

## ­¤źŖ Hard-NAT holepunching

Until this release, iroh could holepunch when the client was behind a hard NAT and the server behind an easy NAT, but not the reverse. With this release, the asymmetric case where the server is behind a hard NAT also works.

See [noq#672](https://github.com/n0-computer/noq/pull/672) and [#4254](https://github.com/n0-computer/iroh/pull/4254) for more details.

## ­¤øŻ’ĖÅ Configurable path selection

For users building on top of [custom transports](/blog/tor-custom-transport), iroh now exposes a `PathSelector` trait that decides which path a connection uses.

```rust
trait PathSelector {
    fn select(&self, ctx: PathSelectionContext<'_>) -> PathSelection;
}
```

`PathSelectionContext` exposes the currently selected path and an iterator over `PathSelectionData`, which today contains the path address and stats. `PathSelection` is a newtype over `Option<Addr>`. Configure a selector via `Endpoint::builder().path_selector(...)`.

The trait and the new types are gated behind the `unstable-custom-transports` feature so they can still evolve after 1.0.

See [#4232](https://github.com/n0-computer/iroh/pull/4232) for more details.

## ­¤öÉ Pluggable relay access control

The relay's access control has been redesigned. The `AccessConfig` enum is gone, replaced by an `AccessControl` trait with `on_connect` and `on_disconnect` hooks. Each connection gets a `ConnectionId` that is passed to both callbacks.

```rust
trait AccessControl {
    async fn on_connect(&self, req: &ClientRequest) -> Access;
    async fn on_disconnect(&self, id: ConnectionId);
}
```

With the `ConnectionId`, an `AccessControl` implementation can track active connections and disconnect clients when their access is revoked, without waiting for them to make another request.

For embedders that mount the relay protocol via their own HTTP server (e.g. axum), the `Bucket` rate-limit primitive is now public. The existing `RateLimited<S: AsyncRead>` wrapper doesn't fit when the layer above the relay handler is already WebSocket-framed; `Bucket` can be applied at the frame layer instead.

See [#4276](https://github.com/n0-computer/iroh/pull/4276) and [#4242](https://github.com/n0-computer/iroh/pull/4242) for more details.

## ­¤¦╣ Stability fixes

- **Faster `Endpoint::close`** ŌĆö Shutdown skips the draining period when it can. Closing is now near-instant when the peer already closed remotely, and roughly one RTT otherwise. Fixes [#4201](https://github.com/n0-computer/iroh/issues/4201) in [#4270](https://github.com/n0-computer/iroh/pull/4270).
- **Remote state races** ŌĆö Two related bugs in the remote map could double-spawn `RemoteStateActor`s or shut one down while `ResolveRemote` requests were still pending. Both fixed in [#4271](https://github.com/n0-computer/iroh/pull/4271) and [#4272](https://github.com/n0-computer/iroh/pull/4272).
- **Path abandonment scope** ŌĆö Abandoning a path on one connection no longer abandons it on every other connection that shares the same address. Other connections will abandon it themselves if it's unusable. Fixed in [#4284](https://github.com/n0-computer/iroh/pull/4284).
- **Static musl builds** ŌĆö A CI fix forces non-PIE `crt1.o` so the static musl release binaries link again. Fixed in [#4258](https://github.com/n0-computer/iroh/pull/4258).

## ­¤ōÉ FourTuple plumbing

Internally the selected path is now tracked as a `FourTuple` (local + remote address) throughout both send and receive paths. The user-visible changes are small:

- `IncomingLocalAddr` has been renamed to `LocalTransportAddr` to reflect that it is used on both sides; the `Relay` variant is now a tuple variant.
- `PathEvent` variants are now `#[non_exhaustive]`.
- Custom transport implementations get a new `src: Option<&CustomAddr>` argument on `CustomSender::poll_send`, matching the source-pinning the IP transport already supports.

See [#4273](https://github.com/n0-computer/iroh/pull/4273) and [#4281](https://github.com/n0-computer/iroh/pull/4281) for more details.
